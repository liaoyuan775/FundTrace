# 首批多来源资金测试材料设计

## 目标

生成一套完全虚构、可重复生成的多来源资金材料，并通过 FundTrace 案件上传和解析接口形成独立案件数据源，用于验证字段提取、证据定位、重复识别和拓扑建模。

## 批次规模

首批固定为 10 份材料：2 份 CSV、2 份 XLSX、2 份文本型 PDF、2 份 Word 表格、1 张手机银行转账截图和 1 张第三方支付账单截图。材料覆盖 32 个虚构实体、120 笔唯一交易和 15 次跨文件重复出现，共计 135 条材料记录。

业务内容覆盖个人银行流水、受害人付款、一级收款、资金分拆、资金归集、资金回流、第三方支付、商户消费和 ATM 取现。不同文件使用不同字段名称、日期格式、金额格式和证据位置。

本批次验证重点是同一批资金记录在多种文件格式下的提取、去重和建图。POS 和 ATM 在本批次中主要体现为渠道及出口实体，不把它解释为已经覆盖真实 POS 签购单或 ATM 凭条版式。独立 ATM 凭条、POS 签购单、商户结算单、余额快照及退款冲正通知单留作第二批，避免首批失败时无法区分文件格式问题与事件语义问题。

## 目录与隔离

生成目录采用批次结构：

```text
test-data/batch-01/
  materials/    # 仅此目录上传到案件
  oracle/       # ground_truth.json、分布清单和预期拓扑，不上传
  reports/      # 上传、解析和对账报告
```

系统新建独立测试案件，不向现有演示案件写入材料或流水。`oracle` 仅供解析完成后的自动对账使用，绝不参与模型输入、草稿生成或拓扑建模。

## 数据流

1. 使用现有合成材料生成器生成 10 份材料及 oracle。
2. 调用案件创建接口生成独立 `case_id`。
3. 通过 `/api/cases/{case_id}/materials` 上传 `materials` 下的文件，生成原始文件和 manifest。
4. 对每个文件调用 `/parse?use_model=true`。结构化表格优先确定性解析，PDF、Word 和图片使用分类及 Qwen 提取。
5. 提取记录写入案件 `drafts/transactions.jsonl`，保留文件、页码、工作表、表格、行号或图片区域。
6. 测试程序读取草稿与 oracle 对账；只有人工确认的 canonical 交易进入拓扑。

仅复制文件到后端材料目录不视为接入，必须走上传接口以形成完整来源链路。

## 失败处理

- XLSX 生成缺少 `@oai/artifact-tool` 时停止并明确报告，不用 CSV 改后缀伪造。
- Qwen 文本或视觉接口不可用时保留已上传材料和结构化解析结果，报告具体失败文件与原因。
- 任一材料解析失败时不删除同案件其他成功记录，案件状态按成功、部分成功或失败统计。
- 解析失败和 Schema 校验失败不得显示为已完成，也不得进入拓扑。

## 验收

- 生成目录中恰好有 10 份规定格式的材料。
- 120 笔唯一交易和 15 次重复出现与 oracle 一致。
- 10 份材料均通过 API 归属于同一测试案件并具有 manifest。
- CSV/Excel 关键字段准确率为 100%；PDF、Word、图片在模型可用时目标不低于 90%。
- 每条草稿至少能定位到一个原始文件位置。
- 去重后的确认拓扑与 oracle 的交易数、金额、节点数和边数一致。
- 输出机器可读对账结果和 Markdown 验收报告，并明确记录模型或运行时环境阻塞。
